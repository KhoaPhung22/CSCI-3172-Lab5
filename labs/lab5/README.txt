Unit Test Description
Server-side tests used supertest to isolate the Express router from the network. Client-side tests used jsdom to isolate UI logic from the live browser environment."
Testing was split into two main components: the server-side API routes and the client-side UI logic.

1. Server-Side Testing (tests/server.test.js):
   The server-side Express routes were tested using Jest and Supertest. To isolate the tests from the actual Spoonacular API and prevent unnecessary API quota usage, `global.fetch` was mocked using `jest.fn()`. This allowed the tests to simulate various Spoonacular responses, including successful recipe data, empty results, upstream 500 errors, and network failures. The application correctly handled missing API keys by returning a 500 error, and invalid parameters (like a non-numeric ID) by returning a 400 error. All server routes worked as expected and handled simulated failures gracefully.

2. Client-Side Testing (tests/client.test.js):
   The front-end utility functions and DOM rendering logic from `main.js` were tested using Jest and JSDOM. The test environment was isolated by simulating the browser DOM, allowing the tests to run headlessly. Functions like `renderRecipeCards`, `addIngredient`, and `stripHtml` were tested extensively for edge cases such as null states, empty search arrays, whitespace-only inputs, and fallback images. A Mock DOM environment was initialized for checking UI elements like the "Empty State" container and modal dialog functionality. All client-side logic successfully passed its assertions and handled invalid outputs appropriately.
